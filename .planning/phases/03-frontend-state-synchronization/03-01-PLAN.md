<architecture_note>
## Loading State Architecture (FRONT-02, FRONT-03, FRONT-04)

The loading states (isStreaming, isSubmitted, isReady) are **automatically managed by the Vercel AI SDK**.

**How it works:**
- `chat.status` is managed by the AI SDK's Chat class
- When `onFinish` fires, the SDK automatically sets status to "ready"
- The derived states in chat-state.svelte.ts (lines 421-424) react instantly:
  ```typescript
  isStreaming = $derived(chat.status === "streaming");
  isSubmitted = $derived(chat.status === "submitted");
  isReady = $derived(chat.status === "ready");
  ```
- `sendMessageEnabled` (line 426-432) depends on `!this.isStreaming && !this.isSubmitted`
- When status becomes "ready", the input is automatically enabled

**Why no implementation tasks for FRONT-02/FRONT-03:**
The AI SDK handles loading state clearing automatically. The onFinish callback does NOT need to manually clear loading state - it happens synchronously when the SDK transitions status to "ready".

**Why FRONT-04 (MCP tools) is covered:**
MCP tool invocations go through the same Chat class and onFinish callback. Tool results are streamed as part of the assistant message. There is no separate loading state for MCP tools - they use the same `isStreaming` derived state. Code Agent operations also use the same Chat class with DynamicChatTransport.

**What this plan fixes:**
The actual issue is race conditions in async operations (title generation, suggestions) that can race with new messages. This plan adds cancellation patterns to prevent stale async operations from corrupting state.
</architecture_note>
